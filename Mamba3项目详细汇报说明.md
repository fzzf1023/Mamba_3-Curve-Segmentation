# Mamba-3 图表曲线实例分割项目详细汇报说明

## 1. 项目概述

这个项目的目标，不是简单地把图表中的所有曲线像素找出来，而是要把图里的每一条曲线都单独分割成一个实例，并尽可能保留它们的连续性、交叉关系和边界关系。项目默认主模型是 `CurveSOTAQueryNet`，训练入口是 `train.py --model sota`，默认配置是 `chart` preset，输入尺寸默认是 `512 x 512`。

如果用一句话概括这个项目，可以这样说：

**它是一个面向图表数字化任务的、基于 Mamba 空间编码器和 Query 解码器的曲线实例分割系统，重点解决细曲线断裂、交叉混淆、相邻曲线粘连和网格线误检问题。**

和普通语义分割不同，这个项目关心的是：

- 哪些像素属于曲线。
- 这些像素分别属于哪一条曲线。
- 一条曲线是否连续。
- 两条曲线交叉时是否仍然能正确区分实例。
- 网格线、刻度线、背景纹理是否会被误认为曲线。

因此，这个项目本质上是“实例分割 + 拓扑建模 + 几何约束 + 结构化后处理”的组合系统，而不是单一的像素分类网络。

## 2. 为什么图表曲线提取是一个难问题

图表曲线提取看起来像一个简单的视觉任务，但实际上它有几个非常典型、而且对通用分割模型很不友好的难点。

### 2.1 细线目标极度稀疏

图表中的曲线通常只占整张图很小的一部分像素，而且很多曲线只有 1 到几像素宽。这样会带来两个问题：

- 正负样本极不平衡，背景远远多于前景。
- 只要预测稍微偏一点，就会造成整条线断掉。

### 2.2 曲线具有长距离连续性

一条曲线往往跨越整张图，局部上看只是一些断续的细边缘，但从全局上看它们是同一条结构。普通卷积更擅长局部纹理建模，而图表曲线更需要“沿着线看很远”，即长程依赖建模。

### 2.3 曲线之间会交叉、贴近、平行

两条曲线可能在局部交叉，也可能在很长一段区域内彼此接近甚至几乎平行。此时如果模型只看局部像素，很容易发生：

- 交叉点处把两条线错接成一条。
- 两条相邻平行曲线被合并。
- 一条线在交叉后被分裂成两段。

### 2.4 图表中有大量干扰结构

常见图表往往包含：

- 网格线。
- 坐标轴。
- 刻度线。
- 图例。
- 文本。

这些结构与曲线一样也可能是线状、细长、有方向性，因此会对普通分割网络造成明显干扰。

### 2.5 图表任务不仅要“看见”，还要“分清”

对图表数字化来说，仅仅得到一张“曲线总区域 mask”是不够的。真正的目标通常是后续数值提取、曲线拟合、图例对应和多曲线分析，因此需要模型不仅找出曲线，还要完成：

- 实例分离。
- 交叉保真。
- 连通性保持。
- 结构一致性恢复。

这就是本项目从一开始就采用“实例级 query 分割 + 像素级拓扑辅助”的根本原因。

## 3. 项目整体流程总览

本项目的整体流程可以概括为 8 个连续阶段：

1. 输入图像进入增强层，生成 `RGB + HSV + Scharr` 多通道输入。
2. 增强后的图像进入 Progressive Mamba Encoder，提取多尺度特征 `f1, f2, f3, f4`。
3. 编码器内部的 Snake Scan 分支对弯曲结构进行可变形扫描，增强曲线连续性建模能力。
4. 多尺度特征通过 FPN 融合，并叠加高分辨率 stem skip 和网格抑制偏置；这两项在当前代码里也都可以独立关闭做消融。
5. 融合特征进入两条分支：
   - Query Decoder 分支负责实例级曲线分割。
   - Pixel Topology 分支负责中心线、交叉、边界、方向、网格等几何拓扑预测。
6. 训练时，Query 输出与 GT 实例通过 Hungarian matching 做一对一匹配，同时结合 O2M 和 DN queries 进行额外监督。
7. 损失函数从分类、mask、质量、中心线、交叉、方向、CAPE、snake offset 等多个维度联合优化模型。
8. 推理时，系统使用质量感知评分、细线恢复、连通域过滤和 crossing/boundary-aware NMS 得到最终实例集合。

因此，这个项目不是“一个 backbone 接一个 mask head”这么简单，而是一整套围绕图表曲线错误模式设计的系统工程。

### 3.1 什么是 H/2、H/4、H/8、H/16、H/32

文档和代码里频繁出现 `H/2`、`H/4` 这类写法，它们表示特征图相对于原图分辨率被下采样后的尺度。

假设输入图像大小是：

- `H x W = 512 x 512`

那么：

- `H/2` 就是 `256 x 256`
- `H/4` 就是 `128 x 128`
- `H/8` 就是 `64 x 64`
- `H/16` 就是 `32 x 32`
- `H/32` 就是 `16 x 16`

在视觉模型中，深层特征通常分辨率更低，但语义更强；浅层特征分辨率更高，但语义更弱。因此，多尺度建模的本质就是在不同分辨率层面分别学习“细节”和“语义”，再把它们融合起来。

### 3.2 什么是 stem、stem_half、f1、f2、f3、f4

这是本项目里最容易让第一次阅读的人困惑的一组名字。

#### 1. stem

`stem` 通常指网络最前面的“输入预处理卷积层”。它的作用不是做复杂语义建模，而是把原始输入先变成更适合后续主干网络处理的特征表示。

本项目的 `CurveMambaEncoder` 把 stem 拆成了两层：

- `stem_layer1`：把输入下采样到 `H/2`
- `stem_layer2`：再把特征下采样到 `H/4`

这样做的好处是：模型不仅能得到后续主干需要的 `H/4` 特征，还能额外保留一个更高分辨率的 `H/2` 特征供后面 skip 使用。

#### 2. stem_half

`stem_half` 就是经过 `stem_layer1` 之后得到的高分辨率特征。它的分辨率是 `H/2`，在默认 `chart` preset 下，通道数是 `dims[0] // 2 = 32`。

如果输入是 `512 x 512`，那么 `stem_half` 的典型形状是：

- `(B, 32, 256, 256)`

它保留了更多边缘、轮廓和局部细节，因此后面会被送到 `stem skip` 中，为 FPN 融合结果补充高分辨率信息。

#### 3. f1、f2、f3、f4

`f1, f2, f3, f4` 是编码器四个 stage 的输出特征图。它们不是任意命名，而是严格对应主干网络四个尺度层级：

- `f1`：Stage 0 输出，分辨率 `H/4`
- `f2`：Stage 1 输出，分辨率 `H/8`
- `f3`：Stage 2 输出，分辨率 `H/16`
- `f4`：Stage 3 输出，分辨率 `H/32`

在默认 `chart` preset、输入 `512 x 512` 时，它们的典型形状是：

- `f1 = (B, 64, 128, 128)`
- `f2 = (B, 128, 64, 64)`
- `f3 = (B, 224, 32, 32)`
- `f4 = (B, 384, 16, 16)`

这四个特征图的角色也不一样：

- `f1` 最接近输入图像，细节最多，语义最浅。
- `f2` 开始具有更稳定的局部结构表示。
- `f3` 兼顾局部和全局，是一个很关键的中层语义特征。
- `f4` 最深，感受野最大，语义最强，但空间细节最少。

因此可以把它们理解成“同一张图的四种观察尺度”：

- 浅层更像显微镜，看到细节。
- 深层更像望远镜，看到整体结构。

### 3.3 什么是 encoder_dims、blocks_per_stage、branches_per_stage

这三个配置名经常一起出现。

#### 1. encoder_dims

它表示每个 stage 的通道数，也就是该尺度特征的“宽度”。

默认 `chart` preset：

- `encoder_dims = (64, 128, 224, 384)`

含义是：

- `f1` 有 64 个通道
- `f2` 有 128 个通道
- `f3` 有 224 个通道
- `f4` 有 384 个通道

通道越多，理论上可表达的特征模式越丰富，但计算量也会增加。

#### 2. blocks_per_stage

它表示每个 stage 堆叠了多少个 `SpatialMambaFusionBlock`。

默认 `chart` preset：

- `blocks_per_stage = (2, 2, 3, 2)`

也就是说：

- Stage 0 有 2 个 block
- Stage 1 有 2 个 block
- Stage 2 有 3 个 block
- Stage 3 有 2 个 block

#### 3. branches_per_stage

它表示每个 stage 内部启用哪些扫描分支。默认 `chart` preset 对应：

- Stage 0：`local + row + col`
- Stage 1：`local + row + col + snake_h + snake_v`
- Stage 2：`local + row + col + row_rev + col_rev + snake_h + snake_v`
- Stage 3：与 Stage 2 相同

它的含义是：不同深度的 stage 使用不同复杂度的方向建模方式，而不是所有层都用同一套最重的分支。

### 3.4 什么是 decoder_dim、embed_dim、out_hw

#### 1. decoder_dim

`decoder_dim` 是 FPN 融合之后统一使用的通道宽度。因为 `f1~f4` 的通道数不同，而后面的 query 解码器和像素头都需要统一通道数的特征，所以要先把多尺度特征投影到同一个维度。

默认 `chart` preset：

- `decoder_dim = 128`

#### 2. embed_dim

`embed_dim` 主要用于 base 模型中的实例嵌入头，表示实例 embedding 的通道数。它和 query 解码器本身不是一个概念。

默认：

- `embed_dim = 16`

#### 3. out_hw

`out_hw` 表示原始输入图像的空间大小，通常就是 `(H, W)`。很多 head 先在较低分辨率上预测，再通过上采样恢复到 `out_hw`，从而与原图对齐。

### 3.5 什么是 fused、mask_feat、memories

#### 1. fused

`fused` 是经过 `FPNDecoder` 和 `stem skip` 之后得到的统一融合特征。它是后续几乎所有预测头共享的核心特征图。

对于默认 `chart` preset，如果输入是 `512 x 512`，由于启用了 `stem skip`，`fused` 的典型形状是：

- `(B, 128, 256, 256)`

也就是 `decoder_dim = 128`，空间分辨率是 `H/2`。

#### 2. mask_feat

在 query 分支里，`mask_feat` 本质上就是用于生成实例 mask 的那张融合特征图。项目里 query 会先产生一个 per-query 的 mask kernel，再与 `mask_feat` 做乘积，得到该 query 对应的 mask。

可以把它理解为：

- `mask_feat` 负责提供整张图的像素特征底座。
- 每个 query 学出一个“如何读取这张底座”的 kernel。

#### 3. memories

`memories` 是给 query decoder 的 cross-attention 使用的多尺度记忆序列。当前实现中它来自：

- `f4`
- `f3`
- `f2`

而不是 `f1`。原因是：

- `f4/f3/f2` 语义更稳定，更适合作为实例级解码的 memory。
- `f1` 分辨率太高、token 太多，直接放入 cross-attention 计算代价很大。
- `f1` 的细节价值已经通过 FPN 和 `fused` 路径保留了。

在进入 decoder 之前，`f4/f3/f2` 会先经过：

- `memory_proj`：1x1 卷积，把通道统一到 `decoder_dim`
- `memory_bottleneck`：轻量瓶颈层，压缩并再投影，减少冗余
- `flatten`：把 `(B, C, H, W)` 变成 `(B, H*W, C)`

所以 `memories` 可以看成：

**“把多尺度语义特征整理成 query 可读取的 token 记忆库”。**

## 4. 网页各部分详细解释

下面按照网页中的章节顺序，对每个部分的基本原理和在本项目中的作用进行详细说明。

## 4.1 Overall Architecture：整体架构

### 这一部分在回答什么问题

它回答的是：整套模型由哪些大模块组成，这些模块之间的输入输出关系是什么，它们分别解决什么问题。

### 基本原理

对于复杂视觉任务，尤其是实例分割任务，通常不会由一个单一模块完成全部工作，而是要分成多个阶段：

- 前端负责提取视觉特征。
- 中间层负责整合局部和全局信息。
- 后端负责输出结构化结果。
- 训练阶段负责设计监督目标和匹配机制。
- 推理阶段负责把原始预测变成稳定可靠的最终结果。

这就是“系统架构”的意义。它并不是数学公式本身，而是对整个求解流程的工程拆分。

### 本项目中的实现

本项目把系统划分为四个核心阶段：

1. 输入增强。
2. Progressive Mamba 空间编码。
3. Query 实例解码 + 像素拓扑头。
4. 结构感知后处理。

这种划分直接对应图表曲线任务的四类主要难点：

- 输入增强对应低对比度和颜色相近问题。
- Mamba 编码对应长程连续性问题。
- Query 解码对应实例分离问题。
- 后处理对应断裂修复、交叉保留和重复抑制问题。

### 为什么这样设计

如果只用一个单一路径去做“曲线/背景”二分类，会有两个明显缺点：

- 模型可以找到曲线区域，但无法明确地区分不同实例。
- 模型即使学到了部分结构信息，也很难在推理阶段显式利用交叉、边界和中心线。

所以本项目采用的是“实例分割主路径 + 拓扑辅助路径”的思路。主路径负责“分开每条曲线”，辅助路径负责“告诉系统这些曲线在结构上长什么样”。

### 汇报时可以这样讲

你可以把整体架构概括为：

**先把图像变成更适合曲线识别的输入，再用 Mamba 建模细长结构的长距离依赖，然后用 query 逐条解码曲线实例，最后结合中心线、边界和交叉信息做结构感知后处理。**

## 4.2 Task Definition, Symbols, and Evaluation Targets：任务定义、符号和评估目标

### 这一部分在回答什么问题

它回答的是：模型到底输出什么，项目如何形式化定义“预测正确”，以及最终评估关注哪些误差。

### 基本原理

一个实例分割任务通常把输出定义为多个实例的集合，而不是单一 mask。对于每个实例，需要至少给出：

- 一个像素级掩码 `M_k`。
- 一个置信分数 `s_k`。

这样就可以把最终结果写成一个集合：

`Y = {(M_k, s_k)}`

如果把所有实例 mask 做并集，就得到最终的曲线总提取结果。

### 为什么图表曲线任务必须这样定义

图表数字化最常见的后续目标是：

- 追踪每条曲线的坐标。
- 对不同曲线赋予不同含义。
- 对应图例。
- 分析交叉和变化趋势。

这些任务都要求输出必须保留“实例身份”，而不是只输出一张“所有曲线合在一起”的图。

### 本项目关注的四类核心错误

网页里把项目重点减少的错误归纳为四类：

- Gap：同一条曲线被断成多段。
- Bridge：两条曲线被错误连接。
- Crossing confusion：交叉点处实例分配出错。
- Grid false positive：网格线被误判为曲线。

这四类错误其实比传统 IoU 更贴近图表数字化任务的实际痛点。

### 评估指标为什么不只看 IoU

IoU 和 Dice 只告诉你像素重叠程度，但图表曲线更需要关注结构和实例关系。因此项目用了多组指标：

- 曲线总提取指标：curve IoU、Dice、Precision、Recall、clDice。
- 拓扑诊断指标：skeleton recall、connectivity、fragmentation、endpoint error。
- 交叉区域指标：crossing IoU、precision、recall、F1。
- 实例分离指标：mAP、PQ、split rate、merge rate、instance-count error。

这些指标共同回答了三个不同层面的问题：

- 曲线有没有提出来。
- 曲线结构是否连续。
- 不同曲线有没有分清。

### 汇报时可以这样讲

你可以强调：

**这个项目的评估体系不是“只要 IoU 高就算好”，而是围绕图表曲线最重要的三个目标构建的：提取得对、结构不断、实例分清。**

## 4.3 Data Labels and Supervision Entry Points：数据标签与监督入口

### 这一部分在回答什么问题

它回答的是：标注数据如何变成训练信号，模型到底学的是什么监督目标。

### 基本原理

深度模型不会直接理解“这是一条曲线”的抽象概念，它只能学习张量形式的监督。对于图表曲线任务，仅有一个二值 mask 往往不够，因为它无法区分以下信息：

- 哪个像素属于哪一条曲线。
- 曲线的中心在哪里。
- 曲线在某个位置的方向是什么。
- 哪些位置是交叉。
- 哪些位置是实例边界。

因此，更好的做法是从折线标注中派生出一组互补的监督图。

### 本项目的数据表示

根据 `dataset.py`，项目会把 polyline 标注栅格化成以下目标：

- `curve_mask`：有宽度的曲线区域。
- `centerline_mask`：1 像素宽的中心线。
- `instance_ids`：每个像素属于哪个实例，0 表示背景。
- `direction_vectors`：主切线方向和交叉次切线方向。
- `crossing_mask`：两条或以上曲线重叠的区域。
- `boundary_mask`：不同实例接触或相邻的边界区域。
- `grid_mask`：网格背景区域，如果有标注则使用真实标签。
- `layering_target`：上下层顺序关系，默认通常未标注。

### 每一种监督的基本原理

#### 1. `curve_mask`

它是最传统的监督目标，表示“这里属于曲线区域”。它主要用于基础的前景/背景区分，让模型先学会哪里有线。

#### 2. `centerline_mask`

它表示曲线的骨架中心。中心线监督有两个优势：

- 它比厚区域 mask 更直接反映“曲线是否连续”。
- 它对线宽变化不敏感，更适合图表这类细线结构。

所以本项目将中心线视为拓扑建模中最关键的监督之一。

#### 3. `instance_ids`

这是实例分割最核心的监督。它告诉模型：哪怕两处像素都属于曲线，它们也可能属于不同实例。Query 分支最终就是围绕这个实例身份来训练和匹配的。

#### 4. `direction_vectors`

它记录曲线局部的切线方向。方向监督的重要性在于：

- 有助于模型区分相交但方向不同的曲线。
- 给 snake offset 对齐提供几何目标。
- 帮助网络理解“曲线该往哪里延伸”。

#### 5. `crossing_mask`

它专门指出“这里是合法交叉区域”。如果没有这个信号，模型很容易把交叉处看成异常重叠，并在后处理时错误压掉其中一条线。

#### 6. `boundary_mask`

它标记实例边界，主要用于加强实例分离。相邻平行曲线虽然颜色和方向可能很像，但它们之间应该存在边界证据。

#### 7. `grid_mask`

它用于告诉模型哪些区域属于网格或背景结构。图表中网格线非常容易干扰曲线检测，因此这个分支的价值在于“主动学会抑制背景周期结构”。

### 为什么需要这么多种标签

因为图表曲线任务本质上不只是一个分割问题，它同时包含：

- 像素分类。
- 实例分离。
- 几何方向建模。
- 拓扑结构建模。
- 背景干扰抑制。

不同监督对应不同任务维度。把这些维度分开显式建模，会比只靠一个 mask 监督更稳定、更可解释。

### 本项目的一个实现细节

当前 stock `dataset.py` 不会自动生成 `legend_patches`，所以图例引导模块在默认数据流下会自动回退到普通 query 路径。这一点在汇报时可以说明：图例模块是“可选增强”，不是硬依赖。

## 4.4 Input Enhancement Layer：输入增强层

### 这一部分在回答什么问题

它回答的是：为什么输入不是普通 RGB，而是 `RGB + HSV + Scharr` 9 通道。

### 基本原理

图像增强并不是为了让图片“更好看”，而是为了给网络提供更适合任务的表示。对图表曲线来说，最重要的两类底层信息是：

- 颜色差异。
- 边缘和方向差异。

RGB 虽然包含颜色，但不够适合直接表达“色相”和“背景亮度分离”；而 Scharr 则能更稳定地给出边缘方向和梯度幅值。

### HSV 的作用

HSV 把颜色分解为：

- H：色相，表示颜色类别。
- S：饱和度，表示颜色纯度。
- V：亮度，表示明暗程度。

它比 RGB 更适合图表场景的原因是：

- 多条曲线亮度可能相近，但色相不同。
- 背景和网格通常饱和度低，曲线颜色往往更鲜明。
- 使用 HSV 可以让网络更容易分离“彩色曲线”和“灰白背景”。

### Scharr 的作用

Scharr 算子会计算图像在水平和垂直方向上的梯度，得到：

- `Gx`：水平方向梯度。
- `Gy`：垂直方向梯度。
- `|G|`：梯度幅值。

这三者本质上是在告诉网络：

- 哪里存在边缘。
- 边缘大致朝什么方向变化。
- 哪些边缘是强边缘，哪些是弱边缘。

对于细曲线来说，Scharr 特征可以让网络在很浅的层级就获得线状结构的几何线索，而且局部梯度响应通常比 Sobel 更稳。

### 为什么增强层特别适合图表任务

自然图像中的目标通常依赖纹理和语义，而图表曲线更依赖颜色对比和几何边界。把 HSV 和 Scharr 直接拼接到输入端，相当于人为给模型加入“颜色分解视角”和“边缘分解视角”。

根据当前实现，`CurveSOTAQueryNet` 会先做：

`images -> RGB + HSV + Scharr -> 9 channels`

再送入主干编码器。

### 汇报时可以这样讲

**输入增强层的目的，是把对图表曲线最关键的底层信息提前暴露给网络：HSV 强化颜色可分性，Scharr 强化细线边缘和方向感知。**

## 4.5 Progressive Mamba Encoder：渐进式 Mamba 编码器

### 这一部分在回答什么问题

它回答的是：为什么主干网络不是纯 CNN，也不是普通 Transformer，而是 Mamba 风格的空间编码器。

### Mamba 的基本原理

Mamba 属于状态空间模型（State Space Model, SSM）的一类实现。它的核心思想是：

- 把输入看成序列。
- 通过递推状态来建模长距离依赖。
- 用比全局自注意力更高效的方式处理长序列。

在自然语言处理中，Mamba 擅长处理很长的 token 序列；在视觉任务中，它可以把二维特征图按某种顺序重排成序列，然后沿序列方向建模长距离关系。

### 为什么图表曲线特别适合 Mamba

图表曲线有一个典型特点：沿着曲线方向的相关性很强，沿着法线方向的相关性较弱。也就是说，真正重要的是“这条线能否从左边持续延续到右边，或从上方延续到下方”。

这与 Mamba 擅长的“长距离连续建模”高度契合。

### 2D 特征为什么还能做序列建模

方法并不是把图片简单拍扁，而是把特征图按不同扫描方向重排成多个序列，例如：

- 按行扫描。
- 按列扫描。
- 反向行扫描。
- 反向列扫描。
- snake 扫描。

这样每个方向都在回答一个问题：如果从这个方向沿序列看，这条曲线是否连续。

### 什么叫 Progressive

Progressive 的意思是渐进式、多阶段地堆叠特征提取，而不是所有层都用同样复杂的分支。

当前默认 `chart` preset 的主干配置是：

- `encoder_dims = (64, 128, 224, 384)`
- `blocks_per_stage = (2, 2, 3, 2)`
- `branches_per_stage = 3 / 5 / 7 / 7`

它对应四个尺度阶段：

- `f1`：H/4
- `f2`：H/8
- `f3`：H/16
- `f4`：H/32

### 为什么深层分支不一味做大

图表曲线场景和通用目标检测不同，实例数通常不算特别多，但结构连续性非常重要。当前实现中，`chart` preset 对深层分支做了任务导向的收敛式设计，避免在最深层保留过多冗余方向分支，从而兼顾：

- 长距离建模能力。
- 参数量和计算量控制。
- 对小实例数图表的适配性。

### 编码器中的稳定训练机制

网页里还强调了几个训练稳定模块：

- `LayerScale`：给残差分支乘一个可学习的小尺度系数，降低早期训练不稳定性。
- `DropPath`：随机丢弃部分残差路径，减少深层共适应，提高泛化。
- `SE Fusion`：为不同方向分支自适应分配权重，避免所有分支一视同仁地混合。

### 汇报时可以这样讲

**Progressive Mamba Encoder 的核心价值，是把“沿曲线方向的长程连续性”显式建模出来，而不是只靠卷积在局部窗口里猜测。**

## 4.6 Snake Scan Branch：蛇形扫描分支

### 这一部分在回答什么问题

它回答的是：为什么项目要引入 snake scan，而不是只用固定的行列扫描。

### 基本原理

固定方向扫描有一个天然问题：真实曲线通常不是水平直线或垂直直线，而是弯曲的、斜向的、形状变化的。若强行按固定方向扫描，序列中的相邻元素未必真的在曲线上相邻。

Snake Scan 的思想是：

- 先预测一个偏移场。
- 再按照偏移后的采样点去取特征。
- 让扫描路径更贴合曲线真实形状。

这本质上是一种“结构感知的可变形扫描”。

### 当前实现如何做

根据 `mamba3_curve_instance_seg.py`，每个 snake 分支都会：

1. 用一个轻量卷积网络预测偏移量。
2. 用 `tanh` 限制偏移范围，避免采样点跳太远。
3. 构造新的采样网格。
4. 用 `grid_sample` 在连续坐标上取样。
5. 将变形后的特征按某个方向重排成序列。
6. 用 Mamba 对该序列建模。
7. 再把结果还原回二维空间。

### 四个方向是什么意思

实现支持四种 snake 扫描方向：

- `horizontal`
- `vertical`
- `diag45`
- `diag135`

但默认 `chart` preset 主要启用：

- `snake_h`
- `snake_v`

而 `legacy` 会额外启用对角方向。

### 为什么要做 offset alignment loss

光预测偏移还不够，因为模型可能学出“有偏移，但偏得没几何意义”。所以项目引入了 snake offset 与 GT 切线方向的对齐损失，基本思想是：

- 横向 snake 的偏移应与曲线的垂直变化相关。
- 纵向 snake 的偏移应与曲线的水平变化相关。
- 对角方向也有各自对应的投影关系。

这相当于告诉网络：偏移不是任意的，而是应该与真实曲线走向一致。

### snake scan 解决的核心问题

它主要解决两个问题：

- 曲线弯曲时固定扫描路径覆盖不足。
- 细线在曲率变化处容易断裂。

### 汇报时可以这样讲

**Snake Scan 相当于让网络“沿着曲线去看”，而不是死板地横着扫或竖着扫。这样对弯曲曲线的连续建模会更自然。**

## 4.7 FPN, Grid Bias, and Pixel Topology Heads：FPN、多尺度融合、网格偏置与像素拓扑头

### 这一部分在回答什么问题

它回答的是：编码器的多尺度特征如何融合，以及像素级的拓扑信息如何被预测出来。

### FPN 的基本原理

FPN 的思想是把深层语义特征和浅层细节特征融合起来。深层特征语义强但分辨率低，浅层特征分辨率高但语义弱。通过自顶向下上采样和横向连接，可以在保留语义的同时恢复空间细节。

对图表曲线来说，这非常重要，因为：

- 曲线很细，需要高分辨率细节。
- 曲线又很长，需要高层语义来识别同一结构。

### 本项目里的 FPN 到底做了什么

当前 `FPNDecoder` 可以拆成 5 个连续步骤理解。

#### 第一步：lateral projection

代码里先对 `f1, f2, f3, f4` 分别做 `1 x 1` 卷积，也就是 `laterals`。它的作用是：

- 不改变空间分辨率。
- 只改变通道数。
- 把不同 stage 的特征统一投影到 `decoder_dim`。

这样四个尺度就拥有了相同的通道宽度，便于相加和拼接。

#### 第二步：top-down pathway

然后 FPN 从高层往低层逐级上采样并相加：

- `p3 = lateral(f3) + upsample(p4)`
- `p2 = lateral(f2) + upsample(p3)`
- `p1 = lateral(f1) + upsample(p2)`

这里的核心思想是：

- 深层的大感受野语义，通过上采样传给浅层。
- 浅层保留自己的高分辨率局部细节。

于是每一层都不再是单独的信息，而是“局部细节 + 上层语义”的组合。

#### 第三步：smooth

相加之后，项目对每个尺度再过一个 `3 x 3` 的 `ConvGNAct`，这就是 `smooth`。它的目的是：

- 消除上采样和相加造成的局部混叠。
- 进一步让同尺度特征更平滑、更一致。

#### 第四步：all_scales 对齐与拼接

接着，项目不是只用最浅层 `p1`，而是把所有尺度都上采样到 `p1` 的分辨率，然后沿通道维拼接：

- `all_scales = [p1, upsample(p2), upsample(p3), upsample(p4)]`

这一步的意义是：

- 不是只做层间逐级传递。
- 而是把所有尺度的结果在同一空间分辨率上显式聚合。

#### 第五步：fuse

拼接后的大张量再经过 `fuse` 模块，用 `1 x 1` 和 `3 x 3` 卷积压回 `decoder_dim`。这一步相当于学习：

- 哪些尺度信息应该保留。
- 不同尺度特征如何在通道维上重新混合。

最终得到的就是 `fused`。

### 什么叫“多尺度特征融合”

所谓“多尺度”，本质上就是同一个目标在不同分辨率、不同感受野下的表示。

在本项目中：

- `f1` 更适合表示局部边缘和细节。
- `f2` 开始表示中尺度结构。
- `f3` 更适合表达较长曲线的局部全局关系。
- `f4` 最擅长表达整条曲线的大范围语义上下文。

“多尺度融合”并不是简单把它们加在一起，而是让：

- 高层语义告诉浅层“哪里更像真正曲线”
- 浅层细节告诉高层“边界和中心线到底落在哪些像素上”

### Stem Skip 的作用

当前实现中还使用了 `H/2` 分辨率的 stem 特征做 skip connection。它的作用是把非常早期的高分辨率边缘信息重新注回解码阶段，防止细线在多次下采样后丢失。

### 什么是高分辨率 stem skip

`stem skip` 是指：

- 先把 `stem_half` 通过 `1 x 1` 卷积投影到 `decoder_dim`
- 再把 FPN 输出上采样到 `H/2`
- 最后把两者相加

写成过程就是：

1. `stem_proj = stem_skip(stem_half)`
2. `out = upsample(fpn_out, size=stem_proj.shape[-2:])`
3. `out = out + stem_proj`

它之所以叫 “high-resolution stem skip”，是因为：

- `stem_half` 来自网络最早期。
- 它的分辨率比 `f1` 还高，是 `H/2`。
- 它保留了极细粒度的边缘和局部对比信息。

### 为什么 stem skip 对图表曲线特别有效

图表曲线往往只有几像素宽。即使 FPN 已经把深层和浅层特征融合了，主融合路径通常仍然主要在 `H/4` 尺度上工作。对自然图像的大目标来说这可能足够，但对细曲线来说，`H/4` 仍可能太粗。

引入 `H/2` 的 stem skip 有两个直接收益：

- 把更高分辨率的边缘定位能力补回来。
- 让最终 `fused` 特征在半分辨率上保留更多细线细节。

如果输入是 `512 x 512`，那么：

- 常规 FPN 主要在 `128 x 128` 上聚合
- stem skip 会把输出提升到 `256 x 256`

这相当于把最终共享特征图的空间精度提高了一倍。

### Grid Bias 的基本原理

图表中常见的网格线是一种周期性的、线状的背景结构。它们很容易激活线状检测分支，因此项目引入了 `GridSuppressionBranch`，估计一个背景偏置图 `B_grid`。

这里有一个重要设计：项目采用的是“加法抑制”而不是“乘法抑制”。

原因在于：

- 乘法门控容易把弱曲线一起压掉。
- 加法偏置更像是在特征空间中加入背景先验，强行告诉网络“这里偏背景”，但不直接清零语义。

这一点特别适合细线任务，因为细线本来信号就弱。

当前实现还把 `stem skip` 和 `grid bias` 都做成了真实结构开关，因此可以分别通过消融验证它们对精度的贡献，而不是只停留在结构图里。

### Pixel Topology Heads 的基本原理

项目的像素头一共输出五类信息：

- `centerline_logits`
- `crossing_logits`
- `boundary_logits`
- `direction_vectors`
- `grid_logits`

它们共同构成像素级的拓扑约束系统。

#### 1. Centerline Head

它负责预测曲线骨架。骨架比厚 mask 更能体现“是否连通”，因此是图表曲线任务中的核心拓扑监督。

#### 2. Crossing Head

它专门学习交叉区域。这样在后处理时，系统就不会把交叉区域简单视为重复重叠。

#### 3. Boundary Head

它学习实例边界，帮助系统分开紧挨着的相邻曲线。

#### 4. Direction Head

它学习局部方向场，帮助网络理解曲线走向，也为 snake 对齐提供几何参考。

#### 5. Grid Head

它学习背景网格区域，辅助抑制背景误检。

### 这些像素头在训练和推理中的不同角色

在训练中，它们主要提供辅助监督，让主干特征更几何化、更结构化。

在推理中，其中三类输出还会直接参与后处理：

- `centerline_logits`
- `crossing_logits`
- `boundary_logits`

这说明这些像素头并不是“训练时有用、测试时扔掉”的附属模块，而是直接影响最终预测质量。

### PredictionHead 是怎么工作的

像素头内部并不是一层裸的卷积，而是先做一次轻量 refinement，再输出最终 logits。

其基本结构是：

- depthwise `3 x 3` 卷积
- pointwise `1 x 1` 卷积
- GroupNorm
- GELU
- 再接 `3 x 3 + 1 x 1` 输出头

这里的含义是：

- depthwise 卷积更高效地做局部空间细化。
- pointwise 卷积负责通道混合。
- GroupNorm 在小 batch 下比 BatchNorm 更稳。

所以这些 head 不是简单“读出层”，而是会先对共享特征做一次专门针对该任务的局部精修。

## 4.8 Query Decoder：Query 解码器

### 这一部分在回答什么问题

它回答的是：模型如何把一张图拆成“若干条独立曲线实例”。

### Query-based 实例分割的基本原理

Query-based 模型的思想来自 DETR、Mask2Former 等框架。它不再像传统方法那样先生成大量候选框或像素聚类，而是直接引入一组固定数量的 query。每个 query 可以理解为一个“实例槽位”：

- 如果某个 query 学到了某条曲线，就输出这条曲线的 mask 和分数。
- 如果某个 query 没有对应实例，就输出背景。

这种方法非常适合实例数相对有限、结构明确的任务。

### 为什么图表曲线适合 Query 分割

图表中曲线实例数通常不算非常大。当前默认配置下：

- `num_queries = 40`

对于大多数“几条到十条左右曲线”的图表来说，这已经足够覆盖实例空间，而且比过多 query 更稳定。

### Query 初始化的基本原理

项目里的 query 初始化不是纯随机的，也不是只靠 learned embedding，而是：

`Q0 = Qlearn + Qcontent`

其中：

- `Qlearn` 提供稳定的可学习实例槽位。
- `Qcontent` 从当前图像内容中池化得到，具备样本自适应性。

这相当于把“模型先验”与“当前图像信息”结合起来，使 query 一开始就不是完全空白的。

### DN Queries 的原理

DN 是 denoising queries。它在训练时用 GT 区域做 masked average pooling，再加一点噪声，构造出额外的 query 锚点。

这样做的好处是：

- 训练早期不容易出现匹配抖动。
- query 不容易全部塌缩到背景。
- 模型能更快学会“一个合理实例 query 应该长什么样”。

### Query Decoder 的层内机制

当前实现中的 Query Decoder 不是简单的标准 Transformer 解码器，而是融合了多种针对图表任务的改造。

#### 1. Top-k Query-Memory Alignment

它的思想是：每个 query 不必对整个 memory 全局关注，而是只对最相关的一小部分 memory token 聚合。

这样有两个好处：

- 减少复杂背景带来的注意力噪声。
- 让每个 query 更专注于自己最相关的曲线区域。

#### 2. Query Self-Attention with Relation Bias

Query 之间不是彼此独立的。两条曲线可能相交、平行、相邻，因此 query 之间存在空间关系。项目使用相对位置偏置来编码 query 参考点之间的：

- `dx`
- `dy`
- 距离
- 角度

这使得 query 自注意力不仅看语义，还显式看空间关系。

#### 3. Cross-Attention with Legend Guidance Bias

在 cross-attention 阶段，query 会去读多尺度 memory。若图例存在，则会把当前的联合图例偏置加入 cross-attention logit。这个偏置不再只有 `LCAB` 颜色项，还会叠加基于结构模板匹配的形状项，并由模态权重自动决定颜色和形状各占多少比例。

#### 4. Query Routing

解码器每层更新后，不是完全替换旧 query，而是用一个门控系数决定“保留多少旧状态，引入多少新信息”。这能减少层间震荡，让 query 更新更稳定。

#### 5. Quality Head

项目为每个 query 额外预测一个 mask 质量分数，而不是只看类别分数。最终排序时使用：

`class_score * quality_score`

这样可以减少“类别很像曲线，但 mask 很差”的伪阳性。

### Query Decoder 输入里的几个关键名字

为了避免汇报时说不清，下面把 decoder 输入和中间变量也解释一下。

#### 1. q 和 q_pos

- `q` 是 query 特征本体，也就是“当前实例槽位内容”。
- `q_pos` 是 query 的位置嵌入，主要用于注意力时给 query 加上位置先验。

#### 2. memory

`memory` 是某个尺度的编码器 token 序列，来自 `memories` 列表中的一个层级。decoder 会轮流读取多个 memory level，而不是死盯一个尺度。

#### 3. ref_points

`ref_points` 是从 query 特征中回归得到的二维参考点，范围在 `[0, 1]`。它不是最终坐标，而是 query 的一个“空间锚点”。Relation Bias 就是基于这些参考点之间的相对位置计算出来的。

#### 4. aux_outputs

`aux_outputs` 是每个中间 decoder layer 的预测结果。它的作用是深监督，也就是：

- 不只监督最后一层 decoder
- 中间层也要尽早学会输出像样的实例预测

这样训练更稳定，梯度传播也更充分。

### memory_proj 和 memory_bottleneck 的意义

Query decoder 使用 `f4/f3/f2` 作为 memory，但它们原始通道数分别是：

- 384
- 224
- 128

为了让 decoder 统一读取，项目先用 `memory_proj` 把它们映射到同一个 `decoder_dim = 128`。再通过 `memory_bottleneck` 做轻量瓶颈变换，减少冗余并提升表达紧凑性。

可以理解为：

- `memory_proj` 解决“规格统一”
- `memory_bottleneck` 解决“表达压缩”

### 为什么 mask 生成是“kernel 乘 feature”

每个 query 最终要输出一张 mask。项目不是为每个 query 单独跑一个大卷积头，而是：

1. 先把 query 通过 `mask_embed` 变成一个向量 kernel。
2. 再让这个 kernel 与 `mask_feat` 做乘积。

代码上对应：

- `kernel = mask_embed(q)`
- `pred_masks = einsum(kernel, mask_feat)`

它的直觉是：

- `mask_feat` 提供整张图的共享像素表示
- query kernel 决定“我要从这张共享特征图里读出哪一条曲线”

这类设计比为每个 query 单独跑完整像素解码器更高效，也更符合 query-based 实例分割的思路。

### BATO 的原理

`BATO` 是第一个解码层中的 memory token 重加权机制。其基本思想是：

- 训练初期 query 对真正目标区域的感知不稳定。
- 若直接做 cross-attention，分类和 mask 分支容易失衡。
- 因此先利用当前 query 对 memory token 做一次相关性重加权，增强可能属于曲线的 token。

可以把它理解为“先把更像曲线的记忆位置突出出来，再让 query 去读”。

### 汇报时可以这样讲

**Query Decoder 负责把一张图分成若干条独立曲线，它不是逐像素聚类，而是让固定数量的实例槽位主动去竞争、匹配和解码各条曲线。**

## 4.9 Legend-Guided Module：图例引导模块

### 这一部分在回答什么问题

它回答的是：如果图表存在图例，系统如何利用图例先验帮助曲线实例分割。

### 基本原理

在很多图表中，图例和曲线之间具有一一对应关系。图例通常提供了非常关键的先验信息：

- 曲线颜色。
- 线型风格，例如实线、虚线、点线。

如果模型能够利用这些先验，就能更容易把视觉相似但语义不同的曲线分开。

### 模块 A：LegendPatchEncoder

这个模块负责把图例 patch 编码成 query 初始化向量。

它没有直接用一个复杂 CNN，而是采用“解析特征 + 小 MLP”的方式，提取两类稳定描述子：

- Lab 颜色统计特征。
- Scharr 结构与周期性描述子。

#### 为什么用 Lab 颜色空间

Lab 颜色空间更接近人类感知，欧氏距离更能反映颜色差异。因此如果图例 patch 和图中某个区域的 Lab 均值更接近，就更可能属于同一条曲线。

#### 为什么用 Scharr 结构描述子

图例不只是“颜色块”，很多时候还带有虚线、点线、点划线和 marker 结构。当前实现会先提取 Scharr 梯度，再结合方向统计和行列投影频谱来描述局部结构：

- Scharr 梯度比旧版 Sobel 更适合细线边缘。
- 方向统计能表示线段朝向和局部形状。
- 行列频谱能保留虚线、点线等周期图案。

这比原先单一的灰度 FFT 风格描述更全面，也更符合现在代码里的颜色/形状双分支设计。

### 模块 LCAB：Legend-Conditioned Attention Bias

它的作用不是只在初始化时用一次图例，而是把“图例颜色与图像位置的相似度”直接加到 cross-attention logits 上，使每一层 cross-attention 都持续受到图例引导。

原理上，它计算：

- 图例 patch 的 Lab 均值。
- 图像在各个 memory 位置的 Lab 颜色。
- 两者的距离。

距离越小，偏置越大，该 query 在 cross-attention 时就越倾向关注该位置。

### 模块 LSAB / LGB：结构偏置与联合引导

除了颜色偏置，当前实现还会构造基于图例结构模板的形状偏置 `LSAB`。它会把图例 patch 变成 Scharr 结构模板，再和主图多尺度特征做归一化相关匹配，突出“形状像这个图例”的区域。

随后，系统不会简单把颜色和形状偏置直接相加，而是通过每个图例的模态权重做融合，形成最终的联合引导偏置 `LGB`。这样：

- 仅靠颜色区分时，颜色项权重大。
- 仅靠形状区分时，结构项权重大。
- 颜色和形状都有效时，两者共同参与。

### 模块 C：Legend Contrastive

这个损失会把“图例编码特征”和“与其匹配的 query 特征”拉近，把不匹配的图例和 query 拉远。它的本质是一种双向 InfoNCE 对比学习。

这样做的意义是：

- 图例先验不只作用于像素层面。
- 它还把 query 的特征空间往“图例可解释”的方向规整。

### 模块 E：Legend Gate

图例并不总是存在，因此系统不能强依赖图例。Legend Gate 通过一个自适应门控将：

- `legend_init`
- `learned_query`

按比例融合。如果图例不存在，或者图例无效，门控就会自动退回到 learned queries。

这个设计的核心优势是：

- 同一个模型可以同时兼容“有图例”和“无图例”两类图表。
- 不需要单独维护两套推理代码。

### 当前实现中的实际状态

当前 `chart` preset 中图例路径是可启用的，但是否真正发挥作用取决于数据管道是否提供 `legend_patches`。如果没有提供，门控会安全回退，且 legend contrastive loss 为零。

### 汇报时可以这样讲

**图例模块的本质是把“图表自带的结构先验”用进模型里：图例告诉模型要找什么颜色、什么线型，模型再利用这些信息去初始化和引导实例 query。**

## 4.10 Matching Strategy and Loss System：匹配机制与损失体系

### 这一部分在回答什么问题

它回答的是：模型训练时如何把预测实例和 GT 实例一一对应起来，以及如何从多个层面对模型进行约束。

### 为什么实例分割需要匹配

Query-based 模型的预测顺序是无序的。也就是说，第 3 个 query 这次可能学到红色曲线，下次可能学到蓝色曲线。因此训练时不能假设“第 i 个 query 永远对应第 i 条 GT 曲线”。

必须先做一个最优匹配，决定：

- 哪个 query 负责监督哪条 GT 曲线。
- 哪些 query 应该学背景。

### Hungarian Matching 的基本原理

Hungarian matching 是一个组合优化算法，用来在预测与 GT 之间找到总代价最小的一对一匹配。

本项目的代价主要由几部分组成：

- 分类代价。
- mask 代价。
- dice 代价。
- 可选 style 代价。

它的意义是让监督不只看类别，也同时看几何重叠质量。

### O2M：One-to-Many 的意义

只做一对一匹配虽然干净，但监督可能过于稀疏。尤其在图表场景中，有些曲线结构复杂，单个 query 很难在早期就稳定学好。因此项目加入了轻量的 O2M：

- 每条 GT 可以分配给少量额外 query 做辅助正样本。

这样做的好处是：

- 提高正样本密度。
- 降低“只靠一个 query 学一条曲线”的训练压力。

### DN Loss 的意义

DN queries 本身也要接受监督，这就是 `dn_mask`。它的作用是让通过 GT 构造的噪声 query 直接学习到正确实例，有助于稳定解码器训练。

### 为什么损失分组

本项目把损失分为 5 大组。这样做不是为了好看，而是因为不同损失的作用对象不同：

- 有的作用于实例级 query。
- 有的作用于像素级拓扑。
- 有的作用于结构连通性。
- 有的作用于几何偏移。
- 有的作用于图例对齐。

### Group A：实例 Query 损失

包括：

- `cls`
- `mask`
- `dice`
- `quality`
- `aux`
- `dn_mask`
- `otm`

这一组负责保证 query 能学会：

- 哪些是前景实例。
- 每个实例的 mask 长什么样。
- 每个实例的质量是否可靠。

这是整个模型的主监督。

### Group B：像素拓扑损失

包括：

- `centerline`
- `crossing`
- `boundary`
- `direction`
- `grid`

这一组负责让特征更“几何化”，而不是只学到粗糙的前景区域。当前默认 `chart` preset 中：

- `centerline = 1.0`
- `crossing = 0.75`
- `boundary = 0.2`
- `direction = 0.25`
- `grid = 0.0`

这说明默认训练策略仍然偏向“连通性和交叉保真”，但不会再让 boundary 完全缺席。

### Group C：Connectivity & Contrastive

包括：

- `cape`
- `pcc`

其中 `CAPE` 是当前默认启用的重要结构损失，`PCC` 在 `chart` preset 中也以小权重默认启用，用来缓解相似 query 之间的混淆。

#### CAPE 的基本原理

CAPE 由两部分组成：

- gap term：要求预测中心线覆盖 GT 骨架，减少断裂。
- bridge term：抑制跨实例边界的错误连接，减少误桥接。

它的本质是同时约束：

- 同一条曲线必须尽量连续。
- 不同曲线不能随便连在一起。

### Group D：Snake Offset

`snake_offset` 用于约束 snake 分支预测的偏移与真实曲线切线方向保持一致。它让可变形扫描路径更有几何意义，而不是纯粹自由漂移。

### Group E：Legend Contrastive

只有当提供 `legend_patches` 时才真正生效，用于把图例特征和 query 特征拉齐。

### 为什么要有手工权重和可选的不确定性加权

网页中给出了两种聚合思想：

- 手工设定权重 `w_i`。
- 不确定性加权。

当前默认 `chart` preset 中更强调人工设定的任务导向权重，因为该任务的重点很明确：连通性、交叉、实例区分，而不是通用分割场景中的泛化平衡。

### 当前实现中的一个关键点：稀疏目标自适应加权

`CurveSOTACriterion` 还实现了 `_task_weight_multipliers`。这意味着每个 batch 中，某些损失会根据目标稀疏程度动态调整。例如：

- 中心线太稀疏时，适当增大 `centerline` 和 `cape`。
- 交叉像素很稀疏时，增大 `crossing`。
- 没有真实 `grid_mask` 时，减弱 `grid`。
- 实例过少时，减小 `pcc` 和 `otm`。

这其实是在解决一个很实际的问题：图表曲线和交叉点天然就是稀疏目标，如果所有损失一视同仁，稀疏结构的监督很容易被背景损失淹没。

## 4.11 Training Strategy and Learning Rate Schedule：训练策略与学习率计划

### 这一部分在回答什么问题

它回答的是：这个多模块、多损失系统如何稳定地训练起来。

### AdamW 的基本原理

AdamW 是深度学习中常用的优化器，它在 Adam 的基础上把权重衰减独立处理，更适合现代视觉模型训练。

本项目默认参数为：

- `lr = 2e-4`
- `weight_decay = 1e-4`
- `grad_clip = 1.0`

### 为什么 Mamba 参数组使用更小学习率

网页说明中指出，Mamba 参数组使用 `0.5 * base_lr`。这是因为状态空间类模块对学习率变化往往更敏感，若学习率过大，序列建模部分容易在训练早期振荡。

因此降低 Mamba 参数组学习率，本质上是在保护这部分更敏感的长程建模模块。

### Warmup 的原理

Warmup 是在训练初期从很小学习率逐步升到目标学习率。其意义在于：

- 刚开始参数还很随机，直接用大学习率容易导致梯度过冲。
- 对多分支结构而言，warmup 可以降低不同子模块之间的训练冲突。

本项目默认：

- `warmup_epochs = 5`

### Cosine Decay 的原理

在 warmup 之后，学习率按余弦曲线缓慢衰减。它的优点是：

- 前期下降较慢，保留足够探索能力。
- 后期下降更平滑，有利于收敛稳定。

### EMA 的原理

EMA 会维护一个模型参数的滑动平均版本：

- 当前参数更新快，但抖动也大。
- EMA 参数更新慢，但更平滑。

在验证和推理时使用 EMA 参数，通常能得到更稳定的预测分数和 mask 质量。

### Loss Ramp 的原理

图表曲线任务中，一些复杂约束例如：

- `boundary`
- `cape`
- `pcc`
- `snake_offset`

如果在训练最早期就以全权重介入，可能会干扰主任务收敛。因此项目会在前若干个 epoch 中逐步增加这些损失的权重。

这相当于：

- 先让模型学会最基础的分类和 mask。
- 再逐渐引入结构和几何约束。

### 为什么这套训练策略很重要

如果没有这套策略，这个模型会面临典型问题：

- query 分支还没学稳，拓扑损失就开始主导。
- Mamba 和普通卷积分支梯度尺度不一致。
- 稀疏交叉和中心线信号在训练早期被背景损失淹没。

因此训练策略并不是附属配置，而是使模型真正可训练的必要条件。

## 4.12 Inference Post-processing, Recovery, and Crossing Preservation：推理后处理、细线恢复与交叉保留

### 这一部分在回答什么问题

它回答的是：模型输出 raw query 后，如何把这些候选变成最终稳定的曲线实例。

### 为什么后处理在图表任务中特别关键

对于一般实例分割任务，只要 query 输出质量足够高，简单阈值加 NMS 就能得到不错结果。但图表曲线任务不同：

- 曲线太细，mask 稍弱就可能被阈值整段删掉。
- 曲线会交叉，普通 NMS 容易误抑制真实实例。
- 曲线可能局部分裂出小片段，不能简单按总像素数过滤。

所以这个项目对后处理做了专门设计。

### 第一步：质量感知评分

最终评分不是只看分类概率，而是：

`score = class_prob * sigmoid(quality)^beta`

这意味着：

- 类别对不对很重要。
- mask 质量好不好也很重要。

这样可以减少“类别分高但掩码很烂”的假阳性。

### 第二步：强 mask 与细线恢复

后处理先使用较严格的 `mask_thresh = 0.5` 得到 strong mask。然后，如果一个 query 的综合分数足够高，并且中心线支持该位置，就允许把较低阈值下的细线像素补回来。

其逻辑是：

- 高分 query 说明整体上可信。
- 中心线支持说明这些像素更可能是断掉的细线，而不是噪声。

这一步对图表任务非常关键，因为细曲线最常见的问题就是 mask 没过 0.5，但骨架信号其实是对的。

### 第三步：分辨率自适应面积阈值

项目不是固定使用一个死的最小像素阈值，而是取：

- `max(min_pixels, H * W * min_area_ratio)`

这样高分辨率图像和低分辨率图像不会用同一套面积门槛，更合理。

### 第四步：连通域过滤

对二值 mask，项目会做小连通域过滤，只保留足够大的连通块，必要时只保留最大若干块。

其基本原理是：

- 真正的曲线实例应该表现为少数较大的连续结构。
- 小碎块往往是噪声。

这比简单按总像素数过滤更符合结构特性。

### 第五步：Crossing-aware and Boundary-aware NMS

这是本项目后处理中最有代表性的部分。

普通 NMS 的思想是：

- 如果两个 mask IoU 太高，就抑制分数更低的一个。

但对图表曲线来说，这会出问题，因为：

- 真实交叉时两条曲线本来就会局部重叠。
- 相邻平行曲线在局部也可能有较高 IoU。

所以项目会在 IoU 超过阈值时，不立刻抑制，而是先查看重叠区域的：

- crossing 置信度。
- boundary 置信度。

若 crossing 证据强，说明更像合法交叉，应放宽抑制。
若 boundary 证据强，说明更像相邻不同实例，也应放宽抑制。

### 本项目后处理的核心思想

它不再把“重叠”简单等同于“重复”，而是进一步判断：

- 这是重复实例？
- 还是真实交叉？
- 还是相邻曲线边界接触？

这使得后处理从“通用 mask 筛选”升级为“结构感知实例判别”。

### 汇报时可以这样讲

**这个项目的后处理不是简单阈值和 NMS，而是利用中心线做断裂恢复，利用 crossing 和 boundary 做结构判别，从而保住真实交叉和细曲线。**

## 4.13 Module Origins, Problem Mapping, and This Design：模块来源、问题映射与设计逻辑

### 这一部分在回答什么问题

它回答的是：为什么系统中会出现这些模块，每个模块到底在解决哪一类错误。

### 基本原理

一个好的方法设计不应该是“把很多流行模块拼在一起”，而应该做到：

- 每个模块都有明确目标。
- 每个模块对应特定错误模式。
- 模块之间互补，而不是重复。

网页里给出的模块映射非常适合汇报时展示方法合理性。

### 错误模式与模块映射

#### 1. 细曲线长程连通性不足

对应模块：

- Progressive Mamba Encoder

原因：

- Mamba 的长程依赖建模能力适合沿曲线连续传播信息。

#### 2. 弯曲轨迹覆盖不足

对应模块：

- Snake Scan
- Snake offset alignment

原因：

- 固定方向扫描很难跟随弯曲线条，snake 可以自适应偏移。

#### 3. 交叉和相邻实例混淆

对应模块：

- Query Decoder
- Relation Bias
- 可选 PCC
- crossing/boundary-aware NMS

原因：

- 实例级建模与结构感知抑制共同处理“交叉不断、相邻不粘”。

#### 4. 网格背景误检

对应模块：

- Grid Bias
- 可选 Grid Loss

原因：

- 通过背景偏置和辅助监督显式压制周期性背景结构。

#### 5. 训练不稳定

对应模块：

- BATO
- O2M
- DN queries
- EMA
- loss ramp

原因：

- 这些模块共同解决 early-stage 匹配抖动、监督稀疏和多损失竞争问题。

#### 6. 图例先验利用不足

对应模块：

- A / LCAB / LSAB / LGB / C / E

原因：

- 图例同时提供颜色和线型先验，对图表任务非常自然。

## 4.14 Terminology Reference：术语与缩写详解

下面对网页中的核心专业名词做集中解释，方便汇报时统一口径。

### SSM / Mamba

状态空间模型。这里用于把二维特征按不同方向重排成序列，再沿序列做长距离建模。它擅长处理细长结构的连续性问题。

### PE

位置编码。由于二维特征会被重排成序列，如果不显式加入位置信息，模型会丢失原始空间坐标语义。

### SE Fusion

Squeeze-Excitation 风格的分支加权融合。它用于动态决定不同方向分支的重要程度，降低冗余分支干扰。

### LayerScale

残差分支的逐通道缩放参数。常用于深层网络稳定训练，避免残差一开始过强。

### DropPath

随机深度。训练时随机去掉部分残差路径，提高泛化能力。

### Snake Scan

可变形方向扫描。通过预测偏移，使扫描路径更贴近真实曲线形状。

### Stem Skip

高分辨率跳连。把早期高分辨率特征送回解码器，补充细节。

### Grid Bias

加性网格背景偏置。学习哪些区域更像网格/背景，并把这种偏置加到融合特征中。

### Memory Tokens

解码器 cross-attention 使用的多尺度编码特征序列，可理解为 query 可读取的“记忆库”。

### BATO

Balance-Aware Token Optimization。先根据 query 相关性增强更有可能属于曲线的 memory token，缓解解码初期分类与 mask 学习失衡。

### Top-k Align

每个 query 只聚合与自己最相关的一部分 memory token，而不是无差别关注全部 memory。

### Relation Bias

根据 query 参考点之间的相对位置构造自注意力偏置，帮助建模交叉、平行和空间关系。

### DN Queries

通过 GT 区域池化和噪声扰动构造出的训练期 query 锚点，用于稳定早期训练。

### O2M

One-to-Many。每个 GT 额外给少量 query 做辅助正样本监督，增强训练信号。

### CAPE

连通性感知损失，由 gap 和 bridge 两部分构成。gap 惩罚断裂，bridge 惩罚跨实例错误连接。

### PCC

Query 级原型对比损失。拉近同一 GT 实例对应的 query，推远不同实例 query。

### LCAB

Legend-Conditioned Attention Bias。用图例颜色和图像位置颜色的相似度来偏置 cross-attention。

### LSAB

Legend Shape Attention Bias。用图例的 Scharr 结构模板与图像局部结构做相关匹配，给 cross-attention 提供形状引导。

### LGB

Legend Guidance Bias。按每个图例的模态权重，把 LCAB 和 LSAB 融合成最终引导偏置。

### Legend Gate

图例自适应门控。图例有效时融合图例 query，图例缺失时回退到普通 learned queries。

### Quality Score

mask 质量分支。用于最终评分时抑制低质量候选。

### clDice

强调拓扑一致性的 Dice，更关注骨架覆盖和结构连通性。

### PQ

Panoptic Quality，综合分割质量和识别质量的指标。

### split_rate / merge_rate

前者表示“一个 GT 被拆成多个预测”的比例，后者表示“多个 GT 被并成一个预测”的比例。

### f1 / f2 / f3 / f4

编码器四个 stage 的输出特征图，分别对应 `H/4`、`H/8`、`H/16`、`H/32` 分辨率。它们共同构成多尺度特征金字塔。

### stem_half

stem 第一层输出的 `H/2` 高分辨率特征图，专门保留下来用于 `stem skip`。

### fused

FPN、all-scale 融合和 stem skip 之后得到的统一特征图，是 query 初始化、像素拓扑头和部分训练辅助路径共享的核心特征。

### memories

由 `f4/f3/f2` 投影、压缩并展平成 token 序列后形成的多尺度记忆库，供 query decoder 的 cross-attention 读取。

### mask_feat

用于实例 mask 生成的共享像素特征图，通常就是 `fused`。

### out_hw

原图的空间大小 `(H, W)`，用于把各个分支的预测上采样回输入分辨率。

### lateral / smooth / fuse

这三个词都属于 FPN 内部：

- `lateral`：1x1 投影，统一通道数
- `smooth`：3x3 平滑细化
- `fuse`：把所有尺度拼接后压回 `decoder_dim`

### p1 / p2 / p3 / p4

虽然代码里没有显式写成 `p1~p4` 变量名，但 FPN 内部列表 `p` 实际上就是各尺度 lateral + top-down 融合后的金字塔特征。

### q / q_pos / query_feats

- `q`：当前 decoder 内部 query 状态
- `q_pos`：query 位置嵌入
- `query_feats`：最后一层 decoder 输出的 query 特征，用于 PCC 等损失

### pred_logits / pred_masks / pred_quality

这是 query 分支三个最核心的输出：

- `pred_logits`：前景/背景分类 logits
- `pred_masks`：每个 query 的实例 mask logits
- `pred_quality`：每个 query 的 mask 质量分数

## 5. 项目的端到端训练流程

为了方便汇报，这里再用步骤化方式，把训练流程完整说一遍。

### Step 1：读取图像与折线标注

数据加载器从图像和 JSON 标注中读取曲线 polyline。然后根据线宽信息，把折线栅格化成各种监督图。

### Step 2：构造监督张量

生成：

- 曲线区域。
- 中心线。
- 实例 ID 图。
- 方向场。
- 交叉区域。
- 边界区域。
- 网格区域。

这一步让任务从“标注点集”变成“神经网络可学习的像素张量”。

### Step 3：输入增强

将 RGB 扩展为 `RGB + HSV + Scharr` 九通道输入，提高颜色区分和边缘感知能力。

### Step 4：编码器提取多尺度特征

图像经过 Progressive Mamba Encoder 和 snake 分支，提取多尺度特征及其长程连续信息。

### Step 5：多尺度融合

通过 FPN、stem skip 和 grid bias 形成用于解码的融合特征。

### Step 6：Query 解码

生成初始 query，结合可选的图例信息和 DN queries，然后逐层做：

- query-memory 对齐
- self-attention
- cross-attention
- FFN
- 路由融合

输出实例级分类、质量和 mask。

### Step 7：像素拓扑预测

并行输出中心线、交叉、边界、方向和网格图。

### Step 8：匹配与损失计算

先用 Hungarian matching 做一对一匹配，再结合 O2M、DN 和多组拓扑损失计算总损失。

### Step 9：优化与稳定训练

通过 AdamW、warmup、cosine、EMA、loss ramp 和 batch 级任务自适应加权来稳定训练。

## 6. 项目的端到端推理流程

同样再用步骤化方式说明推理流程。

### Step 1：输入增强与特征提取

测试图像先经过增强层，再进入编码器和 FPN，得到融合特征。

### Step 2：Query 生成候选实例

解码器输出一组 query 候选，每个候选对应：

- 一个类别得分。
- 一个质量得分。
- 一个实例 mask。

### Step 3：计算综合得分

对每个 query 计算分类分数与质量分数的组合，用于排序筛选。

### Step 4：细线恢复

对高分 query，在中心线支持下把低阈值的潜在细线像素恢复回来。

### Step 5：过滤噪声小块

做连通域过滤，去掉很小的噪声片段。

### Step 6：结构感知 NMS

利用 crossing 和 boundary 证据判断高 IoU 候选究竟是：

- 真重复。
- 真交叉。
- 邻近曲线。

据此决定保留还是抑制。

### Step 7：输出最终实例集合

最终得到若干条曲线实例，每条曲线包含：

- 二值 mask。
- score。

必要时再把它们做并集，得到整体曲线提取结果。

## 7. 这个项目与通用分割模型的关键区别

为了汇报更有层次，建议明确指出本项目和通用模型的不同。

### 7.1 它不是只做语义分割

它做的是实例分割，目标是把每条曲线单独分开。

### 7.2 它不是只靠局部卷积

它依赖 Mamba 的长距离序列建模，去维持细长结构的连续性。

### 7.3 它不是只看 mask

它同时建模中心线、边界、交叉和方向。

### 7.4 它不是通用后处理

它的后处理对图表任务做了专门改造，尤其重视：

- 细线恢复。
- 交叉保留。
- 相邻曲线区分。

### 7.5 它不是只看像素 IoU

它还用结构和实例级指标来评估预测的实用性。

## 11. 更底层的基础算子补充解释

为了保证汇报时不会卡在“这些基础模块到底是什么”上，下面把文档中反复出现但容易默认略过的基础算子也统一解释一下。

### 11.1 1x1 卷积

`1 x 1` 卷积不会扩大空间感受野，它主要做的是“通道重组”。因此它常用于：

- 改变通道数
- 做轻量投影
- 在不改变分辨率的前提下统一特征规格

本项目中它广泛用于：

- FPN 的 lateral projection
- stem skip 投影
- memory_proj
- pointwise 融合

### 11.2 3x3 卷积

`3 x 3` 卷积是视觉网络中最常见的局部空间建模单元。它能感知一个较小邻域内的局部几何结构，因此适合：

- 平滑融合结果
- 细化边界
- 补充局部纹理和边缘关系

在本项目里，`3 x 3` 卷积主要用于：

- FPN 的 smooth 和 fuse
- PredictionHead 的 refinement
- stem 和 downsample

### 11.3 Depthwise 卷积与 Pointwise 卷积

#### Depthwise 卷积

每个通道单独卷积，不做通道间混合。优点是：

- 计算量低
- 适合先做空间细化

#### Pointwise 卷积

通常就是 `1 x 1` 卷积，用来做通道间混合。

把它们串起来就是常见的“depthwise + pointwise”结构：

- 先在每个通道内做空间处理
- 再在通道间做信息融合

这是一种兼顾效率和表达力的轻量设计，本项目很多 refinement 模块都采用了这种思路。

### 11.4 GroupNorm

GroupNorm 是一种归一化方法。与 BatchNorm 相比，它对 batch size 的依赖更小，因此在实例分割、小 batch 训练场景中更稳。

本项目中大量使用 GroupNorm，原因主要有两个：

- 图表任务训练时 batch 往往不大
- 多尺度、多分支结构下，BatchNorm 的统计量更容易不稳定

### 11.5 GELU

GELU 是一种激活函数，比 ReLU 更平滑。它的特点是：

- 对小值不是硬截断
- 梯度更连续
- 在 Transformer、Mamba 和现代视觉网络中使用较多

本项目选择 GELU，本质上是为了让深层和序列建模模块的非线性更平滑。

### 11.6 Bilinear Interpolation

双线性插值是一种常见的上采样方式。它根据周围像素做平滑插值，适合特征图尺寸对齐。

本项目中 bilinear upsample 常用于：

- FPN 的 top-down 上采样
- 多尺度对齐
- 将低分辨率预测恢复到 `out_hw`
- 将 FPN 结果提升到 `stem_half` 的 `H/2` 分辨率

它的优点是平滑、简单、稳定，适合作为 feature-level 尺度对齐手段。

### 11.7 Residual Connection

残差连接就是把输入直接加回输出：

- `y = x + F(x)`

其核心价值是：

- 防止深层网络难训练
- 让网络更容易学习“增量修正”而不是“完全重写”

本项目里的 SpatialMambaFusionBlock、decoder layer 等结构都包含残差思想。

### 11.8 DropPath

DropPath 可以看成“按路径随机丢弃残差分支”。它与普通 dropout 不同，不是随机丢某些神经元，而是随机让一整个残差路径暂时失活。

它的作用是：

- 减少分支之间过度依赖
- 提高泛化能力
- 让不同路径都学会独立承担一部分表示责任

### 11.9 Attention

Attention 的基本思想是：

- query 提出“我在找什么”
- key 告诉系统“每个位置是什么”
- value 提供“每个位置可被读取的内容”

在本项目中：

- self-attention 让 query 之间交流
- cross-attention 让 query 去读取多尺度 memory

如果再加入 Relation Bias 或 LCAB，attention 就不仅看语义相似性，还看空间关系和颜色相似性。

### 11.10 为什么这些基础算子值得讲

因为本项目虽然整体上是一个复杂系统，但它的很多关键效果并不是靠神秘黑盒实现的，而是通过：

- 合理的尺度组织
- 合理的卷积类型选择
- 合理的归一化和激活
- 合理的上采样和残差设计

一步步搭出来的。把这些底层算子讲清楚，听众会更容易理解为什么整套系统既复杂、又是可解释的。

## 12. 所有关键概念的“为什么”版总结

前面的章节已经分别解释了各个模块“是什么”和“怎么做”。这一节专门回答一个更适合汇报和答辩的问题：

**为什么系统一定要这样设计？如果不用这个模块，会出现什么后果？为什么不用更简单的替代方案？**

## 12.1 为什么必须保留多尺度特征，而不是只用最深层特征

只用最深层特征的优点是语义强，但对图表曲线任务有一个致命问题：细线的空间定位会变差。

例如，对 `512 x 512` 输入来说：

- `f4` 只有 `16 x 16`

这意味着一格特征已经对应原图中很大一片区域。对于自然图像中的大目标，这种压缩通常还能接受；但对图表中的细线来说，空间信息会过于粗糙，容易造成：

- 中心线定位不准
- 邻近曲线混合
- 交叉点周围结构模糊

反过来，如果只用最浅层特征，也不行。因为浅层虽然分辨率高，但语义弱，模型可能只能看到“这里有一条线状边缘”，却分不清：

- 这是曲线还是网格
- 这是同一条曲线的延续还是另一条曲线
- 这两条线是交叉还是粘连

因此多尺度特征的必要性在于：

- 浅层负责“像素级位置和边界”
- 深层负责“实例级语义和全局连续性”

少了任意一端，系统都会失衡。

## 12.2 为什么要保留 stem_half，而不是直接从 f1 开始

一个很容易被忽略的问题是：即便 `f1` 已经是最浅层主干特征，它的分辨率也只有 `H/4`。对细曲线而言，这仍可能过于粗糙。

保留 `stem_half` 的原因是：

- stem 第一层最接近原图
- 它仍保留大量早期边缘、颜色对比和局部纹理信息
- 这些信息经过更多下采样和深层融合后会不断被稀释

如果没有 `stem_half`，最终共享特征图只能依赖 `H/4` 甚至更低尺度的信息回推细线位置。其典型后果是：

- 线条边缘发虚
- 很细的局部段落被吞掉
- 预测中心线偏离真实位置

所以保留 `stem_half` 不是冗余，而是在为细线任务保存最后一道高分辨率保险。

## 12.3 为什么 FPN 之后还要再加一个 stem skip

一个常见疑问是：既然已经有 FPN，为何还需要 `stem skip`？

原因在于，FPN 虽然能融合深浅层特征，但它的主融合分辨率通常仍围绕 `f1 = H/4` 展开。FPN 解决的是：

- 多尺度语义如何汇总

而 `stem skip` 解决的是：

- 汇总之后，空间细度还够不够

所以两者不是替代关系，而是上下游关系：

- FPN 先把多尺度语义和细节合成
- stem skip 再把更高分辨率的原始细边缘信息补回来

如果只有 FPN 没有 stem skip，模型仍可能“知道这里大概有一条线”，但在像素层面画不够准；如果只有 stem skip 没有 FPN，模型就会“画得细，但理解不够深”。

## 12.4 为什么 query decoder 的 memory 不直接包含 f1

这也是一个很自然的问题。直觉上，`f1` 最细，似乎最适合解码器读取。但当前实现让 decoder 主要读取 `f4/f3/f2`，没有直接把 `f1` 扁平化成 memory。

原因有两个层面。

### 第一，计算代价

如果输入是 `512 x 512`，则：

- `f1` 分辨率是 `128 x 128`
- 扁平化后就是 `16384` 个 token

对 query decoder 而言，这个长度会显著增加 cross-attention 成本。

### 第二，信息角色不同

`f1` 的最大价值在于高分辨率细节，它更适合作为共享像素底座，通过 FPN 和 `mask_feat` 服务于 mask 生成；而不是直接作为语义 memory 让 query 去做全局读取。

换句话说：

- `f1` 更像“画图底稿”
- `f2/f3/f4` 更像“解码所需的语义记忆”

因此项目选择：

- 用 `f4/f3/f2` 构造 `memories`
- 用 `f1` 通过 FPN 和 `fused` 路径保留细节

这是在精度与效率之间做出的任务导向平衡。

## 12.5 为什么要把 Query Decoder 设计成“固定数量实例槽位”

如果不用 query-based 方法，另一条典型路线是：

- 先做像素分割
- 再做聚类或连通域拆分实例

这种思路在普通场景可行，但在图表曲线中会遇到难题：

- 两条曲线交叉时，单靠连通域很难分实例
- 两条相邻平行曲线靠得很近，像素聚类容易粘连
- 断裂曲线可能被拆成多个片段，后续很难可靠合并

而 query-based 解码的优势在于：

- 先假设“我就要找有限条曲线实例”
- 每个 query 负责竞争一条曲线
- 实例身份在解码阶段就被显式建模

这对图表场景尤其重要，因为图表的实例数通常有限且结构明确。换句话说，query 方法更像“逐条识别曲线”，而聚类方法更像“先找一堆线像素再事后猜怎么拆”。

## 12.6 为什么一个 query 还要同时预测类别、mask 和质量

很多人第一次看会觉得：

- 有类别不就够了吗？
- 为什么还要额外预测 `quality`？

原因是实例分割里常见一种错误：

- query 确实看到了目标
- 分类上判断“这是曲线”
- 但画出来的 mask 很差

如果最终只按分类分数排序，那么这些“看见了但没画好”的候选会被错误保留。加入质量分支后，系统就能把两个维度分开：

- `pred_logits` 回答“这是不是一条曲线”
- `pred_quality` 回答“我画这条曲线画得好不好”

这是一个非常典型、也非常实用的设计。它的根本目的不是让模型更复杂，而是避免“语义正确、几何错误”的候选污染最终结果。

## 12.7 为什么要单独预测 centerline、crossing、boundary、direction

如果只预测最终 mask，模型当然也有可能间接学到这些信息，但问题是：

- 这些信息没有被显式要求学对
- 训练时无法单独约束
- 推理时也无法单独调用

项目把这些拓扑量拆出来单独预测，本质上是把“隐含几何结构”变成“显式监督目标”。

### 为什么要 centerline

因为对细曲线来说，最关键的不是区域有多厚，而是骨架是否连续。如果中心线断了，后续数值提取基本就不可靠。

### 为什么要 crossing

因为交叉区域是图表曲线任务和普通分割任务最大的差异之一。没有 crossing 头，系统很容易把真实交叉误当重复重叠。

### 为什么要 boundary

因为相邻曲线常常颜色接近、距离很近，只靠 mask 相似度不足以区分。boundary 提供了“这里应当分开”的证据。

### 为什么要 direction

因为方向是连接关系的几何先验。方向场能帮助模型理解：

- 曲线接下来应往哪延伸
- 两条线即使靠近，其走向是否属于同一实例

所以这些头并不是“可有可无的辅助项”，而是在把图表任务真正重要的几何条件显式化。

## 12.8 为什么要 Hungarian Matching，而不能固定 query 编号

如果固定规定：

- 第 1 个 query 永远负责第 1 条曲线
- 第 2 个 query 永远负责第 2 条曲线

这在训练中几乎不可行，因为：

- 图中曲线数量不固定
- 曲线排序没有天然一致规则
- 同一个 query 在不同图里可能先学到不同外观模式

Hungarian matching 的价值就在于：

- 不要求 query 有固定身份
- 每次根据当前预测质量动态分配监督

它把 query-based 解码从“人工对号入座”变成“最优匹配监督”，这是这类方法能够成立的关键前提。

## 12.9 为什么还要 O2M 和 DN，而不是只靠 Hungarian

Hungarian 是必须的，但仅有一对一匹配还不够，主要因为监督可能过稀。

### 为什么要 O2M

一条复杂曲线只给一个 query 学，训练早期往往不稳定。O2M 的意义在于：

- 给每条 GT 多一点正样本密度
- 降低主匹配单点失误的代价

### 为什么要 DN

DN query 相当于用 GT 构造训练期锚点，让 decoder 从一开始就看到“像样的实例 query 应该长什么样”。没有 DN 时，query 可能在早期更容易全部塌到背景或产生高噪声匹配。

所以这两者共同回答的是：

**当主匹配过稀、早期训练过不稳定时，如何给 query 解码器补充更稳定的正向引导。**

## 12.10 为什么 CAPE、snake_offset、adaptive weighting 这类损失不是锦上添花，而是任务需要

有时会有人觉得这些损失“看起来很花”，似乎删掉也能训练。但对图表曲线任务而言，它们分别对应非常具体的失败模式。

### CAPE 为什么重要

因为仅靠 mask loss，模型更容易学到“区域覆盖”而不是“结构连通”。CAPE 直接把断裂和误桥接这两个结构错误拉进损失函数。

### snake_offset 为什么重要

因为 snake 分支如果没有几何对齐目标，偏移场可能学成一组对损失有利、但不符合曲线几何的自由采样。offset 对齐损失把它重新绑回真实切线方向。

### adaptive weighting 为什么重要

因为图表里的中心线和交叉本来就是稀疏目标。如果统一损失权重不变，训练时背景和大面积简单区域会在数值上主导优化，稀疏拓扑信号反而学不稳。

所以这些损失设计并不是“为了看起来更先进”，而是在直接解决图表任务的结构性偏差。

## 12.11 为什么后处理不能只用阈值 + 普通 NMS

如果直接：

1. 对 `pred_masks` 做固定阈值
2. 用 IoU 做普通 NMS

那么在图表任务里会出现非常典型的三类问题：

- 弱细线整段消失
- 真实交叉被误抑制
- 相邻曲线被当成重复候选合并

这就是项目为什么要做：

- 中心线辅助恢复
- 分辨率自适应面积阈值
- 连通域过滤
- crossing-aware / boundary-aware NMS

它们共同回答的是：

**如何把“原始候选”变成“结构正确、实例不乱、细线尽量保住”的最终结果。**

## 12.12 为什么 legend 模块必须是“可选增强”，而不是硬依赖

图例确实是图表任务里非常强的先验，但现实数据并不总是满足：

- 图例存在
- 图例位置可裁出
- 图例条目与曲线一一对应

如果把图例模块做成硬依赖，模型会失去泛化性。当前设计把它做成“存在时利用、不存在时回退”的可选增强，意味着：

- 有图例时，模型吃到先验增益
- 无图例时，模型仍能作为普通 query-based 实例分割器工作

这是一种典型的工程稳健性设计，而不是功能妥协。

## 12.13 为什么整个系统必须是“多模块协同”，而不是押宝单一创新点

图表曲线任务的难点不是单点问题，而是多个问题叠加：

- 目标细
- 结构长
- 会交叉
- 背景复杂
- 实例要分开

这意味着不存在一个单独模块能一次性解决全部问题。真正有效的方法往往是：

- backbone 解决长程连续性
- decoder 解决实例身份
- topology heads 解决结构证据
- losses 解决优化方向
- post-process 解决最终结构修正

所以从“为什么”的角度看，这个项目最重要的不是某一个模块本身，而是它把“图表曲线错误模式”分解后，分别用合适手段去解决，再把它们重新整合成一条完整链路。

## 12.14 一句话总结“为什么这个方案成立”

这个方案之所以成立，是因为它没有把图表曲线提取当成普通前景分割，而是把它重新定义为：

- 一个需要长程结构建模的问题
- 一个需要实例级身份管理的问题
- 一个需要显式拓扑监督的问题
- 一个需要结构化后处理的问题

一旦这样定义，Mamba、Snake Scan、Query Decoder、Centerline/Crossing/Boundary、Hungarian/O2M/DN、CAPE 和结构感知 NMS 就都不再是零散模块，而是同一个任务定义下的自然组成部分。

## 13. 老师追问版

这一节不是新的方法说明，而是把答辩或汇报现场最可能被老师追问的问题，整理成一套可直接使用的问答模板。每个问题都尽量按下面三个层次回答：

- 标准回答：适合 20 到 40 秒内说清
- 展开回答：适合老师继续追问时展开 1 到 2 分钟
- 回答重点：提醒你这一题真正要守住的逻辑核心

## 13.1 基础定位类追问

### 追问 1：你这个任务为什么不能直接当成普通语义分割来做？

标准回答：

不能，因为图表曲线任务的目标不是只判断“哪里有曲线”，而是要把每一条曲线单独分出来，并且保留其连续性、交叉关系和实例身份。普通语义分割只输出前景/背景，无法天然区分不同曲线实例。

展开回答：

如果只做语义分割，会有三个实际问题。第一，不同曲线在交叉或靠近时会合成一个整体前景，后续很难再稳定拆开。第二，曲线一旦断裂，语义分割往往只能得到几个碎片，无法知道它们是否属于同一条曲线。第三，图表数字化的下游任务通常需要每条曲线单独提取坐标，因此实例身份不是附加需求，而是主需求。

回答重点：

一定要强调“实例身份”是核心，而不是附属功能。

### 追问 2：为什么你说图表曲线提取比普通分割更难？

标准回答：

因为它同时具有细线、小目标、长程连续、交叉、相邻平行和背景网格干扰这几种困难，而这些困难在普通自然图像分割里很少同时出现。

展开回答：

普通自然图像中的目标通常有面积、有纹理、有区域内部信息，而图表曲线几乎只是一条细长结构。模型一旦局部错一点，就可能造成整条线断裂。再加上交叉和相邻平行曲线，问题就不再是简单的像素分类，而是带有结构推理和实例管理性质的视觉任务。

回答重点：

把“细、长、交叉、相邻、网格干扰”这五个词说出来，老师一般就能接受任务难度设定。

### 追问 3：这个项目最核心的输出到底是什么？

标准回答：

最核心的输出是一组曲线实例 `{mask, score}`，而不是单张总 mask。对于 query 模型来说，还会额外输出 `pred_logits`、`pred_masks`、`pred_quality` 和辅助的中心线、交叉、边界等拓扑预测。

展开回答：

如果从系统角度看，最终有两层输出。第一层是实例输出，也就是每条曲线对应一张 mask 和一个分数。第二层是辅助结构输出，包括中心线、交叉、边界、方向等。这些辅助输出一方面用于训练时显式监督，另一方面在推理时用于后处理，因此不是无用副产物。

回答重点：

不要只说“输出是 mask”，要说“输出是实例集合 + 结构辅助图”。

## 13.2 架构设计类追问

### 追问 4：为什么要用 Mamba，而不是纯 CNN 或标准 Transformer？

标准回答：

因为图表曲线的关键难点是长距离连续性。纯 CNN 更擅长局部感受野，标准 Transformer 虽然也能建模全局关系，但在高分辨率视觉场景下成本较高。Mamba 适合把二维特征沿扫描方向转成序列，再高效建模长程依赖。

展开回答：

如果用纯 CNN，模型更容易学到局部边缘，但不容易稳定维持“沿着一条曲线看很远”的连续性。Transformer 确实可以建模全局，但对图像高分辨率 token 做全局自注意力代价较大。Mamba 介于两者之间，既保留长程序列建模能力，又更高效。对图表曲线这种“沿结构方向连续”的目标，它比纯局部卷积更自然。

回答重点：

别只说“Mamba 更先进”，要说“它更贴合细长结构的长程建模需求”。

### 追问 5：为什么是 `f1/f2/f3/f4` 四层，而不是更多层或更少层？

标准回答：

四层是一个任务导向的折中。层数太少，多尺度信息不足；层数太多，会增加复杂度且对小实例数图表不一定有收益。四层已经能覆盖从细节到全局语义的主要尺度范围。

展开回答：

当前默认主干的层级分别是 `H/4`、`H/8`、`H/16`、`H/32`。这已经足以形成一个从高分辨率细节到低分辨率全局语义的稳定金字塔。如果继续增加更深层，一方面 token 更少、空间细节更弱，另一方面计算代价更高，而且图表曲线任务并不像大场景检测那样特别依赖极深层语义。因此四层是围绕该任务而不是围绕通用网络深度做的选择。

回答重点：

强调“这是围绕图表任务的尺度需求做的折中”。

### 追问 6：为什么要用 FPN？直接把最深层特征上采样不行吗？

标准回答：

直接上采样最深层特征会损失大量细节。FPN 的意义是把深层语义和浅层细节在多个尺度上逐步融合起来，避免只靠粗糙高语义特征恢复细线位置。

展开回答：

最深层特征最大的优势是“知道这是不是曲线”，但最大的缺陷是“画不准曲线在哪个像素上”。细曲线任务对像素级定位极其敏感，因此不能只靠深层上采样。FPN 通过 lateral、top-down、smooth、all-scale fusion，把不同尺度的语义和细节聚合起来，这样最终共享特征才既懂结构、又懂位置。

回答重点：

记住一句话：FPN 解决的是“语义够强但定位不准”的问题。

### 追问 7：为什么 FPN 之后还要再加一个高分辨率 stem skip？

标准回答：

因为即使经过 FPN，主融合分辨率通常仍然围绕 `H/4` 展开，而图表细曲线对空间精度要求更高。`stem_half` 提供了 `H/2` 的高分辨率细节补充。

展开回答：

FPN 已经很好地做了多尺度语义整合，但它主要还是在金字塔层之间做汇总。图表曲线很多时候只有几像素宽，`H/4` 仍可能不足以稳定保住细边缘。因此项目把更早期的 `stem_half` 保留下来，通过 skip 的方式直接注回最终共享特征图。这样做的本质是给细线保留更强的空间定位能力。

回答重点：

这题要讲清 FPN 和 stem skip 不是重复，而是“语义融合”和“空间精度补偿”两个层次。

### 追问 8：为什么 query decoder 读取 `f4/f3/f2`，不直接读取 `f1`？

标准回答：

因为 `f1` 的 token 数太多，直接做 cross-attention 成本高，而且 `f1` 更适合作为共享像素底座保留到 `mask_feat` 路径中。`f2/f3/f4` 更适合作为语义 memory。

展开回答：

对 `512 x 512` 输入来说，`f1` 对应 `128 x 128`，如果直接展平成 memory，会有 16384 个 token。对 query decoder 而言，这会显著增加 cross-attention 的代价。同时，`f1` 的价值主要在于高分辨率细节，而不是成熟的实例语义。项目因此让 `f1` 通过 FPN 服务于 `fused/mask_feat`，而让 `f2/f3/f4` 负责实例级 memory 读取。这是效率和信息角色的双重考虑。

回答重点：

一定要同时说“计算代价”和“信息角色不同”。

## 13.3 Query 与实例解码类追问

### 追问 9：为什么要用 query-based 实例分割，而不是做完像素分割再聚类？

标准回答：

因为图表曲线在交叉、贴近和平行时，仅靠后续聚类很难稳定拆分实例。query-based 方法在解码阶段就显式维护实例身份，更适合该任务。

展开回答：

像素分割后再聚类的思路，本质上是先得到一堆“曲线像素”，再尝试事后把它们拆成实例。但图表场景里最困难的地方恰好是实例关系，例如交叉点、近邻平行线和断裂重连。query 方法相当于一开始就给模型若干“实例槽位”，每个 query 主动竞争一条曲线，这比后验聚类更稳定，也更适合与 Hungarian 匹配、质量分支和结构辅助结合。

回答重点：

说清“实例身份应该前置到解码阶段，而不是后置到聚类阶段”。

### 追问 10：为什么默认设置 40 个 queries？这个数怎么理解？

标准回答：

40 不是拍脑袋定的，而是基于图表曲线实例数通常有限这一任务先验做的容量设定。它既要覆盖常见图表中的曲线数量，又不能让 decoder 因过多 query 产生不必要的计算和噪声。

展开回答：

query 的数量本质上是“实例槽位容量”。太少会出现多个 GT 曲线争抢同一个槽位；太多则会导致大量冗余 query 参与训练和排序，增加背景竞争和推理成本。图表场景通常不是几十上百个实例，而是几条到十条左右曲线，因此 40 个 query 在该任务上是相对充足而不过度保守的默认值。它还可以按数据集规模继续调节。

回答重点：

把它解释成“容量超参”，而不是神秘常数。

### 追问 11：为什么 query 还要有 `quality` 分支？

标准回答：

因为分类正确不代表 mask 质量高。质量分支是为了区分“看见目标”和“画好目标”这两件事。

展开回答：

图表曲线里很常见一种情况：某个 query 的确在语义上捕捉到了曲线，因此分类分数不低，但由于局部断裂或边界漂移，生成的 mask 很差。如果最终只按分类分数排序，这类候选就会混进结果里。质量分支给了系统第二个判别维度，使后处理排序更稳，也更符合实例分割任务的真实需求。

回答重点：

记住这句话：`pred_logits` 判断“是不是”，`pred_quality` 判断“画得好不好”。

### 追问 12：为什么 decoder 里还要有 `aux_outputs` 深监督？

标准回答：

因为如果只监督最后一层 decoder，中间层容易学习不足，梯度也更难稳定传播。深监督能让每层 query 都尽早学会有意义的实例预测。

展开回答：

多层 decoder 的一个常见问题是：前几层可能只是给最后一层打工，本身不学可用表示。加入 `aux_outputs` 后，每一层都要输出分类和 mask，从而被显式监督。这通常会带来两个收益：第一，训练更稳定；第二，query 的中间演化过程更有方向感，不容易到最后一层才突然“纠偏”。

回答重点：

把深监督解释成“让每层都承担学习责任”。

### 追问 13：BATO、Top-k Align、Relation Bias 这些模块是不是太复杂了？

标准回答：

它们不是随便叠加的复杂化，而是在分别解决 decoder 里的三个具体问题：早期记忆读取失衡、query 对 memory 关注过散、query 之间缺乏显式空间关系建模。

展开回答：

`BATO` 解决的是 early-stage memory token 重要性不平衡，帮助第一层 decoder 更快把注意力集中到目标区域。`Top-k Align` 解决的是 query 面对海量 memory 时注意力太散的问题，让每个 query 更专注于与自己最相关的 token。`Relation Bias` 解决的是 query 之间本来存在交叉、平行和相邻等空间关系，但普通自注意力并不会显式知道这些关系。三者分别对应三个不同弱点，不是简单堆料。

回答重点：

老师问“是不是复杂了”，你要回答“复杂不是目的，问题对应才是目的”。

## 13.4 拓扑监督、训练与损失类追问

### 追问 14：既然最终输出是 mask，为什么还要预测 centerline？

标准回答：

因为对细曲线来说，最重要的不是区域多厚，而是骨架是否连续。centerline 比普通 mask 更直接刻画“不断裂”这个目标。

展开回答：

如果只用区域 mask 监督，模型更容易学到“把曲线附近区域覆盖住”，但未必能保证骨架连续。尤其对细线任务，哪怕区域重叠还不错，也可能在骨架上出现断点。centerline 把这个问题显式拉到监督目标中，从而直接推动模型学习结构连通性，而不是只学区域覆盖。

回答重点：

强调 centerline 是“结构监督”，不是普通前景监督的重复。

### 追问 15：为什么 crossing 和 boundary 要分别建模？不能一个头一起学吗？

标准回答：

不能简单混成一个头，因为两者表达的是不同结构语义。crossing 表示“这里允许重叠”，boundary 表示“这里应当分开”。

展开回答：

这两个概念在推理决策中甚至是方向相反的。crossing 强时，说明高 IoU 可能是合法交叉，不应轻易 suppress；boundary 强时，说明高 IoU 更可能是相邻实例接触，也不应简单合并。虽然它们都与“实例关系”有关，但在语义上一个偏向允许共存，一个偏向强调分界，因此最好分别建模。

回答重点：

一句话：crossing 是“合法重叠证据”，boundary 是“实例分开证据”。

### 追问 16：为什么必须用 Hungarian matching？

标准回答：

因为 query 没有固定顺序，不能预设第几个 query 对应第几条曲线。Hungarian matching 用最优匹配把无序预测和无序 GT 对齐。

展开回答：

query-based 模型里，预测的排列是动态的。如果没有匹配机制，就相当于要求网络自己在整个训练过程中维持一个固定 query 身份，这是非常不现实的。Hungarian matching 让监督不依赖固定编号，而只依赖当前预测与 GT 的匹配质量，因此是 query 实例解码能够成立的前提。

回答重点：

老师要的是“为什么必须匹配”，不是“匈牙利算法怎么实现”。

### 追问 17：为什么除了 Hungarian 还要 O2M 和 DN？

标准回答：

因为一对一监督太稀疏，训练早期容易不稳定。O2M 增加辅助正样本，DN 提供更稳定的 query 锚点，两者共同提升训练稳定性。

展开回答：

Hungarian 的优点是干净，但缺点是每个 GT 只强监督一个 query。对于复杂曲线，这种监督可能过于稀疏。O2M 的价值是让每条 GT 多分配少量 query 作为辅助学习者；DN 的价值是利用 GT 直接构造噪声 query，让 decoder 更快知道“好的 query 应该长什么样”。一个补密度，一个补稳定性。

回答重点：

这题核心是“单靠一对一监督太稀疏”。

### 追问 18：为什么需要 CAPE，而不是只靠 mask 和 dice loss？

标准回答：

因为 mask/dice 更关注区域重叠，而 CAPE 直接关注图表曲线任务真正敏感的结构错误：断裂和误桥接。

展开回答：

在图表场景中，一个模型即使区域 IoU 不低，也可能在中心线上出现断裂，或者把两条曲线错误连起来。对下游数字化来说，这种错误比少一点边缘像素更致命。CAPE 把 gap 和 bridge 直接写进损失，等于告诉模型：你不仅要覆盖住区域，还要保证连通关系对。

回答重点：

不要只说“CAPE 是 topology loss”，要说它具体在惩罚哪类结构错误。

### 追问 19：为什么有些损失在 `chart` preset 中权重是 0？

标准回答：

因为默认配置不是“全部堆满”，而是按任务优先级选择主监督。对于默认图表场景，中心线、交叉、CAPE 和实例解码更核心，某些辅助项可以保留实现但不强制启用。

展开回答：

一个成熟系统不应该默认把所有可选损失都打开。不同任务的数据分布、实例数和标注丰富度不同。`chart` preset 的思想是：保留所有能力，但默认只启用最契合图表曲线主目标的监督项。这样训练目标更聚焦，避免辅助损失过多相互竞争。需要做消融或更重配置时，再切到 `legacy` 或单独开启相关项。

回答重点：

关键逻辑是“保留能力，不等于默认全开”。

## 13.5 推理、评估与局限类追问

### 追问 20：为什么后处理不能省略？难道不是端到端网络就够了吗？

标准回答：

理论上端到端网络已经给出候选，但图表细曲线任务对结构错误非常敏感，后处理负责把“原始候选”修正为“结构可用结果”，尤其是在细线恢复和交叉保留上很关键。

展开回答：

端到端模型输出的是一组 query 候选，它们仍然会包含弱 mask、局部碎片、重复实例和交叉误判。自然图像实例分割里，简单阈值和 NMS 往往足够；但图表曲线任务里，弱细线和真实交叉如果不做结构感知后处理，很容易被误删。后处理不是在替代模型，而是在把模型的辅助结构输出真正利用起来，完成任务闭环。

回答重点：

一定要讲“后处理是在使用模型已经学到的结构信息”，不是硬编码补丁。

### 追问 21：为什么评估指标不能只看 IoU 或 mAP？

标准回答：

因为图表数字化更关心结构是否可用，而不仅是像素是否重叠。IoU 和 mAP 不能充分反映断裂、交叉保真和分裂/合并错误。

展开回答：

举个典型例子，一条曲线被断成两段，IoU 可能仍然不算特别差，但对后续坐标恢复几乎是不可接受的。再比如两条曲线在交叉点被错误合并，整体前景覆盖可能还不错，但实例层面已经失败。因此项目增加了 `curve_connectivity`、`curve_fragmentation`、`crossing_f1`、`split_rate`、`merge_rate` 等指标，让评估更贴近真实应用目标。

回答重点：

强调“可用性”和“像素重叠”不是一回事。

### 追问 22：你这个方法最适合什么场景？最不适合什么场景？

标准回答：

它最适合多条细曲线、存在交叉或相邻关系、需要实例级提取的图表。相对不适合的场景是：曲线数量极多、图例与曲线关系混乱、背景噪声极端复杂或者标注分布与训练集差异很大。

展开回答：

当前默认 `chart` preset 本身就是围绕“图表曲线实例数不算特别大、但结构要求高”的场景设计的。如果曲线数量非常夸张，query 数量和匹配策略可能都要重新调。如果图表风格与训练数据差异很大，例如存在大量手绘线、极端低分辨率或非常特殊的图例布局，也可能需要重新微调输入增强、图例路径或后处理阈值。

回答重点：

不要回避局限性。能说清适用边界，老师反而更认可。

### 追问 23：如果把这个系统做消融，你认为最值得先验证哪几块？

标准回答：

最值得优先验证的是四块：输入增强、Mamba + snake 主干、query 解码增强模块、结构感知后处理。因为它们分别对应颜色/边缘、连续性、实例管理和最终结构修正四条主线。

展开回答：

如果做论文式消融，可以按“问题导向”拆：

- 去掉 HSV/Scharr，看细线和低对比场景是否下降
- 去掉 snake scan，看弯曲连续性是否下降
- 去掉 centerline/crossing/boundary 辅助，看交叉和断裂是否恶化
- 去掉 BATO/Top-k Align/quality，看 query 解码质量是否变差
- 去掉结构感知后处理，看最终 split/merge 和 crossing 指标是否下降

这样每个消融都能对应一个明确的错误模式，不会变成机械地“删模块看分数”。

回答重点：

要体现你理解实验设计，而不是只背模型结构。

### 追问 24：如果没有 legend，这个方法会不会失效？

标准回答：

不会。图例模块是可选增强，不是硬依赖。没有 `legend_patches` 时，系统会自动回退到普通 query-based 实例分割路径。

展开回答：

项目专门设计了 `Legend Gate` 和条件启用的 legend contrastive loss，就是为了避免图例成为强依赖条件。有图例时，它提供额外颜色和线型先验；无图例时，系统仍然依赖 query decoder、拓扑头和后处理完成任务。这种设计兼顾了先验利用和工程鲁棒性。

回答重点：

这题一定要说“可选增强，不是双模型切换”。

### 追问 25：如果老师说“你这个方法是不是太复杂了，不够优雅”，该怎么答？

标准回答：

可以回答：复杂度不是目标，问题分解才是目标。图表曲线任务本身就是由多个结构性难点叠加而成，所以需要多个互补模块协同解决；但这些模块并不是无序堆叠，而是各自对应明确错误模式。

展开回答：

你可以进一步强调，本方法的复杂性是“任务驱动复杂性”，而不是“为了追求新颖度而加模块”。如果老师关心工程实用性，还可以补一句：当前实现已经提供 `chart` preset，通过更轻的配置把复杂系统收敛为更适合图表场景的默认版本，这说明方法不是一味做重，而是在任务目标和计算成本之间做了实际权衡。

回答重点：

答辩时不要为“复杂”辩解，而要把复杂重新解释成“针对复合难点的分解求解”。
